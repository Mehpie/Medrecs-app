AI:
This breakdown of Self-Correcting RAG (CRAG) and Prompt-Driven Knowledge Graphs (KG) covers the essential loop needed for high-reliability AI systems. Below is a detailed look at how these mechanisms function to ensure the AI doesn't just "guess," but actively validates and corrects its own logic.

1. The Core Philosophy: "Verify, then Trust"

In standard RAG, the system is a one-way street: Retrieve → Generate. If the retrieval is bad, the generation is hallucinated.

Self-Correcting RAG introduces a feedback loop. It treats the retrieved data as "candidates" rather than "truth" until they are graded.

2. Deep Dive: The CRAG Mechanism

The "Corrective" part of CRAG relies on a Grading Layer that sits between retrieval and generation.

The Grader: Usually a smaller, faster model (or a Cross-Encoder) that asks: "Does this chunk actually contain the answer to the user's prompt?"

The Decision Tree:

Confident: If chunks are highly relevant, proceed to synthesis.

Ambiguous: If the internal knowledge is "fuzzy," the system triggers a Query Rewrite to search the Vector DB again with better keywords, or it hits a Web Search API to find up-to-date info.

Irrelevant: The system discards the noise and loops back to the Planner to try a different retrieval strategy (e.g., switching from Vector search to Graph search).

3. Hierarchical Retrieval (Coarse-to-Fine)

To handle complex data, the system uses a Parent-Child indexing strategy.

Coarse Chunks (Parents): Large blocks of text (e.g., an entire legal section) that provide broad context.

Fine Chunks (Children): Small, precise sentences or data points (e.g., a specific date or name) embedded within that parent.

The Benefit: The system can find the exact "needle" (Fine) while still understanding the "haystack" (Coarse) it came from.

4. The Auditor: The Hallucination Gatekeeper

The Auditor is the final safety check. Even if the RAG chunks are good, the LLM might still misinterpret them during synthesis.

Claim-Source Mapping: The Auditor breaks the draft answer into individual claims.

NLI (Natural Language Inference): It checks if Claim A is logically entailed by Source Chunk A.

The "Back-to-Planner" Trigger: If a claim isn't supported, the Auditor sends the task back to the Planner with a specific error message: "Hallucination detected regarding the shipping date; re-retrieve case records for 2023."

5. Prompt-Driven Knowledge Graph (KG)

The KG adds a layer of Relational Logic that vector databases lack.

Neighborhood Extraction: Instead of just finding "similar" text, the Planner looks for Entities and their Relationships (e.g., Contract -> Signed_By -> Person_A).

JIT (Just-In-Time) Triples: If the graph is missing a link, the system can extract a "Triple" (Subject-Predicate-Object) from a text chunk and "UPSERT" it into the graph. This allows the system to "learn" and grow its structured memory based on every successful user interaction.

Summary Checklist for Implementation
Component	Key Metric for Success
Planner	How well it identifies when to use Graph vs. Vector.
Grader	Precision in filtering out "noise" documents.
Auditor	Sensitivity to subtle hallucinations (e.g., wrong dates/names).
Redis	Efficiency in managing the state without bloating memory.

This architecture ensures that the system is resilient. If the first attempt at an answer fails a check, the system has the "agency" to try again, search elsewhere, or refine its query until it meets the required threshold of accuracy.

AI:
To get from the raw, unstructured output of Document AI to the highly structured, forensic JSON you need for your multi-agent system, you have to bridge a specific gap.

Document AI's Layout Parser gives you spatial structure (it tells you where the paragraphs, headers, and tables are on the page), but it does not give you semantic structure (it doesn't know that a specific sentence is a "Diagnosis" or a "Patient History").

To get your final structured data, you pair Document AI's output with an LLM using a technique called Structured Information Extraction. Here is the exact pipeline you need to build:

Step 1: Parse and Chunk (Document AI)

You run the 2,000 pages through the Layout Parser. The output you get is a large JSON file containing the raw text, broken down by pages and blocks.

Your code's job here: Iterate through Document AI's response and split the text into manageable chunks (e.g., 1-2 pages at a time) while keeping the page_number attached to every chunk as metadata.

Step 2: Define a Strict JSON Schema (The Blueprint)

Before you send these chunks to your LLM agents (like Vertex AI's Gemini or OpenAI's GPT-4o), you must define the exact structure you want back. You do this using a JSON Schema or a tool like Pydantic in Python.

For your medical-legal report, you would define different schemas for your different agents. For example, the Orthopedic Agent might have this schema:

Python
from pydantic import BaseModel, Field
from typing import List, Optional

class MedicalFinding(BaseModel):
    date_of_service: str = Field(description="The date the medical exam occurred (YYYY-MM-DD).")
    provider_name: str = Field(description="Name of the doctor or clinic.")
    diagnosis: List[str] = Field(description="List of formal medical diagnoses.")
    pain_level: Optional[int] = Field(description="Reported pain level out of 10.")
    page_source: int = Field(description="The page number this data was found on.")

Step 3: LLM Extraction with "Structured Outputs"

Modern LLMs have a feature specifically designed for this called Structured Outputs (or Function Calling). You send the text chunk from Step 1 and the schema from Step 2 to the LLM, with a prompt like:

"You are an expert medical-legal extraction agent. Read the following text chunk. Extract any orthopedic diagnoses and pain levels. You MUST return your response as a valid JSON object matching the provided schema. Do not include any other text."

Because you are using Structured Outputs, the LLM will not reply with conversational text. It will return a perfectly formatted JSON string:

JSON
{
  "date_of_service": "2023-04-14",
  "provider_name": "UW Medical Center",
  "diagnosis": ["Moderate degenerative disc changes", "Calcific tendinosis"],
  "pain_level": 9,
  "page_source": 670
}

Step 4: Validate, Store, and Synthesize

As your extraction agents process all the chunks from the 2,000 pages, they will generate hundreds of these small JSON objects.

Validate: Use your code (Pydantic) to verify that the LLM didn't hallucinate data types (e.g., ensuring pain_level is actually a number).

Store: Save all these JSON objects into your database (like PostgreSQL or MongoDB).

Synthesize: Now that your data is perfectly structured and categorized in a database, your final Forensic Agent can easily query it. You can programmatically ask your database: "Get me all JSON objects where the diagnosis involves the shoulder, sort them by date, and send them to the Synthesis Agent to write the summary."

Summary of the Tech Stack

Initial Parsing: Google Cloud Document AI (Layout Parser)

Extraction Logic: LangChain or LlamaIndex (Python frameworks for agent routing)

Data Structuring: Pydantic (Python) to define the JSON schemas.

Extraction Engine: Gemini 1.5 Pro or GPT-4o (configured to use Structured Outputs/JSON mode).

To seriously reduce costs—especially if you plan to scale this beyond a single 2,000-page file—you have to look at the entire pipeline. While the Document AI cost is relatively low ($20 for 2,000 pages), the LLM token costs will be the most expensive part of this pipeline. Feeding 2,000 pages of text into GPT-4o or Gemini 1.5 Pro will burn through millions of tokens quickly.

Here are the best architectural strategies to drastically cut costs while maintaining medico-legal accuracy:

1. Implement "Model Tiering" (Don't use Pro for everything)

You do not need a flagship model to extract dates from a billing ledger.

The Grunt Workers (Cheaper Models): Use Gemini 1.5 Flash or GPT-4o-mini for the Extraction Agents. These models are exceptionally fast, strictly follow Pydantic JSON schemas, and cost a fraction of a penny compared to their larger counterparts. Use them to extract the raw facts (dates, ROM degrees, lab values).

The Brain (Flagship Models): Only use Gemini 1.5 Pro or GPT-4o for the Final Forensic/Auditor Agent. Let the cheap models do the reading, and pay the premium only for the final medico-legal synthesis and causation reasoning.

2. Triage and Filter (The "Bouncer" Agent)

In a 2,000-page medical file, up to 40% of the pages are useless (fax cover sheets, duplicate billing disclosures, blank pages, or privacy notices).

The Fix: Before sending a chunk to the expensive extraction agents, run a fast, local script (using a free library like PyPDF2 or pdfplumber) to grab the raw text.

Send that raw text to a highly aggressive, low-cost classifier prompt: "Is this page a fax cover sheet, a privacy notice, or blank? True/False." If True, drop the page. You instantly save 30-40% on downstream OCR and extraction costs.

3. Use Asynchronous Batch APIs (50% Discount)

Since this is an asynchronous Celery task and you likely don't need the 2,000-page summary generated in 10 seconds, you should use Batch APIs.

Both OpenAI and Google offer Batch APIs where you upload a massive JSONL file of requests and get the results back within 24 hours.

The Benefit: Batch APIs typically offer a 50% discount on token costs compared to synchronous API calls. This is a massive saving for large document processing.

4. Hybrid OCR (Free vs. Paid)

Document AI Layout Parser is great, but you might not need it for every page.

Digitally Native PDFs: If parts of the record are digitally generated (you can highlight the text with your mouse), use a free Python library like unstructured.io, PyMuPDF, or Marker to extract the text and tables for $0.00.

Scanned/Messy Pages: Only route the images, handwritten doctor's notes, and scanned faxes to Google Document AI.

5. Context Compression

LLMs charge by the token. Medical records are full of repeating boilerplate (e.g., the hospital's address, the doctor's NPI number, and standard disclaimers printed at the bottom of every single page).

The Fix: Write a simple Python regex (Regular Expression) script to strip out this repeating boilerplate before sending the chunk to the LLM. Shaving 100 tokens of useless footers off 2,000 pages saves 200,000 tokens per run.

The Optimized Cost Flow

Free/Cheap Routing: PyMuPDF reads the file. It drops cover sheets. It extracts text from digital pages for free. It sends only scanned pages to Document AI.

Batch Extraction (Cheap): The parsed text is batched and sent to Gemini 1.5 Flash via the 50%-off Batch API to populate the JSON schemas.

Synthesis (Premium): The resulting JSON database is sent to Gemini 1.5 Pro to write the final Medico-Legal case.

what is this data

AI:
This data is a structured evaluation rubric or grading matrix used to weigh the evidentiary strength of various medical diagnostics, clinical methodologies, and provider types. Specifically, it is tailored for assessing Traumatic Brain Injury (TBI), concussions, and related complex multi-system disorders (neurological, vestibular, and autonomic).

Given the context of building a medico-legal extraction pipeline, this dataset acts as the "Rulebook" or "Knowledge Graph Ontology" for your Forensic/Auditor Agent. It teaches the AI how to value different pieces of medical evidence when building a legal case.

Here is a breakdown of how the data is structured and what it represents:

1. The Structure of the Data

If this were formatted as a table, the columns would likely be:

ID: (e.g., 1, 2, 3...)

Weight/Score: (e.g., 100, 95, 90, 85, 82, 78, 72) This is a critical metric. It assigns a numerical value to the strength of the evidence.

Category: The broad grouping of the evidence type.

Item/Entity: The specific test, methodology, or provider (e.g., Diffusion Tensor Imaging (DTI)).

Justification/Description: Why this item matters legally and medically (e.g., FDA-cleared advanced neuroimaging; detects white matter microstructural injury).

Tags/Domain: The medical discipline (e.g., Neurology / Imaging; Critical_Appraisal).

2. The Grading Hierarchy (The "Weights")

The data establishes a clear hierarchy for what constitutes strong vs. weak evidence in a clinical or legal setting:

Score 100: Gold-Standard FDA-Cleared Objective Diagnostics

These are the heaviest hitters. They include unarguable, machine-generated objective data like DTI, 3T MRI, VNG (Videonystagmography), and Autonomic Reflex Screens.

Score 95: Systems-Integrative / Functional Providers

This grades the source of the opinion. It gives high weight to specialized providers who look at the whole body (Board-certified functional neurologists, FABBIR, multidisciplinary concussion clinics) rather than isolated specialists.

Score 90: Evidence-Based Function-Specific Diagnostics

Highly reliable objective tests that might lack the broad "Gold Standard" FDA-cleared status for TBI specifically, but are functionally critical (e.g., qEEG, VEP, Buffalo Concussion Treadmill Test, ambulatory blood pressure monitoring).

Score 85: Triangulation / Pre-Post / Repeated-Trial Evidence

This evaluates the methodology of the doctor. Did they confirm a finding using two different tests? Did they do a pre-treatment and post-treatment comparison? (This is heavily used in legal defense to prove an injury isn't a fluke).

Score 82: Comprehensive Neuropsychological Assessment

Gold-standard cognitive testing (WAIS-IV, WMS-IV) performed by board-certified neuropsychologists, specifically highlighting the importance of "performance validity testing" (to prove the patient isn't faking).

Score 78: Multi-System Hub-Edge Methodology

Grades whether the medical record integrates different bodily systems (e.g., linking eye movements, neck pain, and dizziness together).

Score 72: Differential Diagnosis Formation

This is the defense against alternative theories. It awards points if the doctor explicitly ruled out other causes of the symptoms, such as pre-existing migraines, Sleep Apnea, PTSD, or Long COVID (which directly parallels the Atarah Phillips case).

How to Use This in Your AI Architecture

Cache-Augmented Multi-Document Causal Extraction System and method

with Relational Vector Indexing, Controlled Vocabulary Enrichment,

and Citation Provenance




Background

Technical Field

The present disclosure relates

generally to computer-implemented systems and methods for processing

and analyzing data originating from bulk documents, and more

particularly, relates to computer-implemented systems and methods

for processing and analyzing data originating from bulk documents in

cache-augmented generation architecture using large language models

(LLM).

Description of the Related Art

Existing systems used for medical

document analysis include LLM pipelines, retrieval-augmented

generation (RAG) systems, and human expert review. However, these

current systems suffer from several technical limitations when

applied to multi-document analysis, such as medical-legal analysis.

Legacy document summarization

systems that are configured to produce descriptive summaries of

medical records do not generate causal narratives linking findings

across temporally separated documents, do not cross-reference

generated claims against published medical literature, and do not

produce page-level citations grounded in vector-space distance. Such

legacy systems produce catalog-style summaries of individual records

rather than cross-document causal analysis.

For one, these known systems rely

on fragment-level retrieval, which loses cross-document causal

context and cannot scale with increasing complexity of raw data. RAG

systems retrieve chunks based on semantic similarity to a query but

do not maintain relational awareness between entities or objects

that appear in different documents or on distant pages. Traditional

review methods process information sequentially or linearly, that

is, document by document and record by record, one item at a time.

Typically, these linear review layers produce summaries,

AI-generated or otherwise. These traditional linear methods are

capable of organizing data by date or source but miss cross-record

relationships. For example, a causal relationship between a COVID-19

history recorded in a billing note and a brain fog symptom

documented in an examination note three years later cannot be

identified by these current systems, which retrieve each document

independently.

In the use case of medical related

litigation, thousands of pages of medical records describe events

but fail to clarify how those events connect to injury, causation,

or outcome. A system is needed to reveal the critical relationships

hidden inside unstructured documentation that standard linear review

fails to capture and that may be exploitable by opposing counsel.

Moreover, various disciplines rely on document analysis to prepare

detailed, citation-supported reports for use in contexts such as

investigations, litigation, and expert testimony. However, existing

systems, including general-purpose large language models, are known

to generate inaccurate or fabricated citations, known as

hallucinations, resulting in unreliable reports. Without a mechanism

that preserves page-level provenance through the vectorization and

generation process, language models produce outputs that cannot be

automatically verified by the system against source documents. Such

outputs may be inadmissible in legal proceedings. Prior

citation mechanisms in LLM systems rely on keyword matching,

retrieved chunk provenance, or direct quotation. A

Furthermore, standard embedding

systems discard structural metadata altogether. Off-the-shelf text

embedding pipelines convert document text to vectors but do not

encode structural metadata (page number, document type, or other

necessary and useful contextual information, such as clinical or

legal entity class) as queryable relational properties. This makes

it impossible to retrieve or identify which page of which document

contains a finding causally linked to a particular symptom.

Lastly, another known drawback of

legacy systems is that single-document analysis misses longitudinal

causation, almost by definition. Medical-legal cases require

analysis across years of records from multiple providers, yet

current systems are limited by their single-document structure and

fail to process, let alone recognize, trends or causation over time.

A system is needed that is able to address these and other needs.

Brief Summary

Briefly stated, the present

disclosure provides a cache-augmented multi-document causal

extraction system and method that (1) receives and ingests a batch

of cache-augmented multiple documents, (2) causally extracts and

organizes data from the cache-augmented multiple documents while

preserving section-specific metadata, (3) maps relationships with

the organized data using relational vector indexing (4) encodes

relational properties into causal-candidate pairs, (5) determines

page-level citation-supported provenance for the causal-candidate

pairs, and (6) produces a citation-supported output reflecting

cross-document causal relationships found within the data.

In some embodiments, the

cache-augmented multi-document causal extraction system employs a

custom extraction and vectorization pipeline that encodes extracted

text units along with their respective structural document metadata

and domain-specific entity tags. In another aspect of some

embodiments, the system constructs a relational cache using a

Cache-Augmented Generation (CAG) architecture that operates on a

full relational vector representation of all of the multiple

documents simultaneously rather than retrieving fragments. In still

another aspect of some embodiments, the system employs an external

large language model that uses engineered prompts to perform causal

reasoning across the relational cache. In yet another aspect of

some embodiments, the system applies a controlled vocabulary of

domain-specific forced tags to enrich the generated output and

identify causal-candidate entity pairs. Furthermore, in another

aspect of some embodiments, the system assigns page-level citations

to every claim in the report by computing vector-space distances

between claim embeddings and page-level entries in the relational

cache.

Standard

Cache-Augmented Generation (CAG) systems enable context-aware

generation but do not support causal traversal, that is, the ability

to follow a causal-candidate relationship across temporal and

documentary gaps. Without relational properties encoding

causal-candidate pairs across the full document set, a CAG system

cannot identify that an entity in one document is causally linked to

an entity in another document separated by years of records.

Distance-based citation ranking systems that assign citations by

computing semantic similarity between a query and document fragments

do not apply domain-specific dual entity encoding, do not perform

causal analysis, and are not designed to produce legally defensible

reports. Such systems rank citations by surface-level semantic

proximity rather than by causal-relational grounding in a structured

document cache.

Cache-Augmented

Generation is an alternative to Retrieval-Augmented Generation (RAG)

for single-document or small-context tasks. However, the present

system provides the technological improvement of applying CAG

architecture to multi-document and multi-disciplinary analysis that

correctly identifies significant relationships amongst the different

domains. Critical insights do not reside in individual

records or isolated data points but rather in the network of

connections between them. The relationships truly represent the

intelligence. Additionally, in some embodiment, the cache-augmented

multi-document causal extraction system has an architecture that

enables relational mapping to process bulk documents or records

relationally.

Additionally,

in one or more embodiments, the cache-augmented multi-document

causal extraction system produces sophisticated reports with

accurate citations to case-specific documents and also sources cited

that originate from outside the provided documents. Thus, the

cache-augmented multi-document causal extraction system maintains a

unified relational representation of all documents simultaneously,

enabling causal reasoning across the full patient history.

In some

embodiments, the cache-augmented multi-document causal extraction

system includes relational mapping, pattern recognition, and

scalable intelligence. The relational mapping of the system

transforms isolated records into structured network models. These

models establish relationships between entities, events, findings,

and outcomes across an entire dataset simultaneously. The pattern

recognition of the system identifies causal pathways,

inconsistencies, and meaningful signals that emerge only when

information is examined as a connected system rather than a linear

sequences of documents and/or events. The scalable intelligence is

an analytical computing system built to operate across large volumes

of fragmented information. This fragmented information may include

analytical computing system produces coherent insight regardless of

scale or complexity.

Methodology

The system 100 carries out a method

of creating relational intelligence from fragmented data,

transforming unorganized information into a network model capable of

producing actionable insight across any domain. The method moves

through six operations as described in detail below.

Batch Document Ingestion and

Extraction 510

In one embodiment shown in FIG. 5,

documents concerning the same case, event, or instance are uploaded

simultaneously as a batch of documents 102 to the cache-augmented

multi-document causal extraction system 100 by a user via a user

terminal. The system 100 accepts heterogeneous document types

including: Subjective, Objective, Assessment, Plan (SOAP) notes,

Electroencephalography (EEG) reports, Magnetic Resonance Imaging

In some embodiments, a custom extraction pipeline of the

cache-augmented multi-document causal extraction system 100 is

This metadata preservation at

extraction time provides a technological improvement for subsequent

operations such as data manipulation and generating reports 108 with

accurate citations to metadata elements including, e.g., correct

page numbers. The extracted text units having preserved metadata is

a technological improvement over current insufficient systems where

standard PDF extraction pipelines discard page metadata during text

extraction. The extraction pipeline of the cache-augmented

multi-document causal extraction system 100 explicitly retains and

encodes the respective document metadata as a structured attribute

of each extracted text unit.

Custom Vectorization with Entity

Encoding 520

Next in the embodiment shown in

FIG. 5, the cache-augmented multi-document causal extraction system

100 vectorizes the extracted metadata-text data or metadata-text

units from the batch of documents 102 to generate embeddings or

vector representations of the data that encode both the semantic

content of the text and its associated structural metadata. The

resulting vector representation for each text unit includes at least

one of the following: a semantic embedding of the text content;

domain-specific or disciplinary entity tags, (e.g., (1) clinical

entity tags include symptoms, diagnoses, medications, diagnostic

findings, anatomical references, temporal markers, and other similar

standard clinical categories, or (2) legal entity tags include

injury events, causation claims, functional limitations, daily life

impact descriptors, and other similar industry-standard useful or

helpful legal categories); and structural metadata, e.g., page

number, document type, source document ID, and section identifier.

The combination of semantic

embedding with industry-specific or domain-specific entity tagging

at the vectorization stage is key for the cache-augmented

multi-document causal extraction system 100 to subsequently identify

significant relationships amongst the data. This combination of

semantic content and named entities is an improvement over the

current known systems because standard medical Natural Language

Processing (NLP) pipelines encode either semantic content or named

entities, but not both simultaneously in a form that supports

relational querying. By encoding both in the same vector

representation, the system 100 enables queries of the form, for

example: “find all clinical findings of type [X] that are

structurally associated with legal entities of type [Y] across all

documents in this case.”

Relational Vector Cache

Construction 530

In another aspect of the embodiment

shown in FIG. 5, vectorized or embedded units are stored with

relational properties in a suitable vector database or embedding

database configured to store vector representations. In some

embodiments, the database is a Qdrant-based vector database. The

relational cache is constructed to support multi-variable relational

querying across the full batch of documents 102. No document in the

batch of documents 102 is left unrelated in order to provide a

comprehensive analysis of the entire evidence records.

In some embodiments, the database

schema encodes relationships between vector entries along specified

dimensions, such as temporal, causal candidate, cross-document,

cross-modal, page-provenance, and other similar dimensions useful to

the discipline and user. A temporal relationship is an ordering of

findings by date of documentation across all source documents. A

causal candidate is an individual entity tag belonging to an entity

class that the controlled vocabulary recognizes as a potential

participant in a causal relationship. A causal-candidate pair is the

confirmed combination of two causal candidates whose entity classes

have an established causal relationship as defined by the controlled

vocabulary, that is, where one entity class is a known causal

precursor of the other. The controlled vocabulary is the authority

that determines which entity class combinations constitute valid

causal-candidate pairs. Without a controlled vocabulary entry

confirming the relationship, two causal candidates remain unrelated.

A cross-document relationship links findings in different source

documents belonging to the same patient record. A cross-modal

relationship links findings from different diagnostic modalities

(e.g., EEG finding → clinical symptom). A page-provenance or

page-attribution relationship involves bidirectional mapping between

each vector entry and its source page number and document

identifier.

These relationships or relational

structures constitute the “multi-variable mindmap” that is

necessary for the generative model to operate in this unique

relational way. The construction of this relational cache is what

distinguishes the approach of the cache-augmented multi-document

causal extraction system 100 from both standard RAG (which retrieves

independently) and standard CAG (which caches without relational

structure). The relational properties of the relational cache enable

the system 100 to identify causal relationships across the batch of

documents 102 rather than only semantic similarities.

Insight

or Inference on Cached Relational Structure 540




Referring again to the embodiment shown in FIG. 5, a large language

model 104 that is suitable to the specific use-case or domain is

used as the generative component or module of the cache-augmented

multi-document causal extraction system 100. In at least one

embodiment, Google’s MedGemma language model

104, which is optimized for medical comprehension, is used in the

medical domain. The large language model 104 operates on the full

cached relational structure of the relational cache rather than just

on retrieved fragments. In this Cache-Augmented Generation (CAG)

architecture, the large language model 104 has access to the

complete relational representation of all documents 102 in the case

during the inference process, rather than receiving only the top-k

semantically similar chunks as in standard RAG.




In one or more embodiments, the large language

model 104 is directed by a set of engineered prompts that are

hard-coded in the codebase of the system 100. These engineered

prompts instruct the large language model 104 to perform specific

reasoning tasks over the relational cache built from the batch of

documents 102. These tasks include: (1) longitudinal analysis by

identifying how findings evolve across time within the full document

set; (2) causal relationship identification by identifying which

earlier findings are causally implicated in later documented

symptoms, drawing on both the relational structure of the relational

cache and the medical knowledge of the model 104; (3) objective

metric extraction that surfaces specific quantitative findings

(scores, measurements, test results, etc.) with their source

citations; and (4) direct quotation to produce verbatim quotes from

source documents with page-level provenance.




The causal output of the cache-augmented multi-document causal

extraction system 100 is not produced by prompting alone. If a large

language model 104 was presented with the same prompts without the

relational vector cache, it would produce generic causal reasoning

based on its training data, not patient-specific causal chains

grounded in the specific documents 102 of the case.




The causal relationships identified by the large language model 104

emerge from the intersection of three components operating together:

(1) the relational metadata encoded in the relational cache (which

identifies causal-candidate entity pairs); (2) the engineered

prompts (which direct the model to reason causally over the cache

structure); and (3) the large language model’s domain knowledge

(which validates whether a causal candidate pair is plausible). No

single component produces the causal output. The novel output

emerges from the combination of these three components.

Additionally, some embodiments of

the cache-augmented multi-document causal extraction system 100

include fine-tuning the large language model 104 on a specific

dataset of medical-legal document that pairs with verified causal

annotations. This fine-tuning further improves causal identification

accuracy beyond the current base-model-plus-prompting approach.




Controlled Vocabulary Forced-Tag Enrichment 550




In another aspect of the embodiment shown in FIG. 5, some domains

include a technological improvement from the use of a

domain-specific vocabulary. After the large language model 104

generates its initial first output over the relational cache, the

cache-augmented multi-document causal extraction system 100 applies

a domain-specific or controlled vocabulary enrichment layer. This

vocabulary enrichment layer includes a predefined taxonomy of

domain-specific concepts or “forced tags” that are

systematically applied to the generated output to identify gaps and

opportunities for enrichment. The controlled vocabulary taxonomy

defines which entity combinations are valid causal-candidate pairs

for forced-tag application. This taxonomy is unique to the relevant

field of the use case and is a custom-made classification system

built for the specific domain.




In some embodiments, the forced-tag enrichment operates as follows.

The cache-augmented multi-document causal extraction system 100

scans the first output to identify entity tags present in the

generated causal claims. For each identified entity tag, the system

100 queries the controlled vocabulary to determine whether a valid

causal-candidate pair exists for that entity. Where the controlled

vocabulary identifies a valid causal-candidate pair that is absent

from or underrepresented in the first output, the system 100 flags

that pair as a “gap” and generates a supplemental causal claim

or enrichment entry directed to that pair. The system 100 then

queries external database 106 for supporting information

corresponding to each flagged causal-candidate pair. Claims

supported by retrieved literature are retained and enriched with

citations. In contrast, claims for which no supporting literature

is retrieved are flagged in the enriched output so that the user may

exercise independent judgment as to their inclusion in the final

report 108. The result of controlled vocabulary forced-tag

enrichment is the enriched output, which comprises the causal claims

of the first output as supplemented and validated by the controlled

vocabulary and supporting literature retrieved from the external

database 106, for inclusion in the report 108.

The controlled vocabulary is

domain-specific and is not publicly available or derived from a

public ontology. In the medical-legal domain, the controlled

vocabulary taxonomy is necessary in order to draw useful,

significant, and valid connections between clinical and legal

domains. This legal-medical taxonomy specific to traumatic brain

injury (TBI) and personal injury (PI) cases was developed by a

board-certified functional neurologist (FACFN, FABBIR) and encodes

the domain expertise of a practicing specialist in functional

neurology and brain injury rehabilitation. The taxonomy defines

which clinical entity combinations are medically valid causal

precursor/sequela pairs for legal documentation purposes. In other

embodiments, the taxonomy covers additional medical specialties

beyond functional neurology and TBI. The taxonomy is the bridge

between the relational structure of the relational cache and sources

outside of the original document upload 102, such as the external

database 106.




Vector-Space Distance Mapping for Page-Level Citation 560

Referring

again to the embodiment shown in FIG. 5, the next stage of the

pipeline produces the cited, legally defensible output. For each

claim or finding in the enriched output, the cache-augmented

multi-document causal extraction system 100 computes the

vector-space distance between the claim embedding and all page-level

vector entries in the relational cache. Notably, the relational

cache was built from the batch of documents 102. The page entry with

the minimum distance (highest semantic similarity) in vector space

is assigned as the citation for that claim,

along with its source document identifier and page number.

This mechanism produces citations

that are grounded in semantic similarity to the source material

rather than in keyword matching or positional proximity. For

example, a claim about a causal relationship between COVID-19

history and subsequent cognitive symptoms is cited to the page in

the source documents 102 where the most semantically relevant

documentation of that relationship exists regardless of whether the

same keywords appear on that page.

FIG. 6 illustrates another method

executed by the cache-augmented multi-document causal extraction

system 100 to produce the report 108 from the batch of documents

102. At operation 610, the system 100 receives, via a computing

device, a batch of documents uploaded by a user. At operation 620,

the system 100 converts, using an extraction pipeline, each document

of the batch into structured text units preserving, for each

structured text unit, associated metadata comprising at least one a

source document identifier and a page number. At operation 630, the

system 100 embeds the structured text units in a manner that encodes

semantic text content, metadata, and domain-specific entity tags.

At operation 640, the system 100 constructs a relational cache by

storing the embedded structured text units in a vector database with

relational properties encoding temporal, causal-candidate,

cross-document, and page-provenance relationships among the vector

representations.

At operation 650, the system 100

performs Cache-Augmented Generation (CAG)

inference by prompting a large language model using

engineered prompts to analyze the relational cache and generate a

first output comprising causal claims derived from relationships

among vector representations across the batch of documents. At

operation 660, the system 100 compares the first output to a

controlled vocabulary of predefined valid causal entity pairs of

more than one causal-candidate, retrieving supporting information

from one or more third-party databases for each identified pair, and

generating a second output. At operation 670, the system 100

computes, for each claim in the second output, a vector-space

distance between a claim embedding and page-level vector

representations in the relational cache, and assigning a page-level

citation to each claim based on a minimum computed distance.

Finally, at operation 680, the system 100 generates a report

comprising the second output and the page-level citations.

Demonstrated Example Use in

Medical Domain of Longitudinal Causal Analysis

The following describes a

representative test case demonstrating the ability of the

cache-augmented multi-document causal extraction system 100 to

identify causal relationships across temporally and documentarily

separated clinical findings. Specific patient identifiers and case

details have been anonymized.

The input material included in a

merged medical record batch of documents 102 include one or more of

the following document types: (1) SOAP notes spanning multiple

clinical visits over a multi-year period; (2) billing records from

primary care and specialist encounters; (3) EEG reports from

neurological evaluation; (4) MRI imaging notes; and (6) examination

notes from neurological and functional assessments. In some

embodiments, the total document length is within the 700-1,200 page

range characteristic of TBI/PI cases processed by the system 100.

The cache-augmented multi-document

causal extraction system 100 identified a causal chain linking a

documented COVID-19 infection history (appearing in early clinical

records) to a subsequently documented brain fog and cognitive

impairment symptom cluster (appearing in neurological evaluation

records approximately three years later).

This causal

identification required: (1) recognition of the COVID-19 history

entry as a clinical entity tag of type [infectious disease /

systemic inflammatory event] in the early billing and SOAP records;

(2) recognition of the brain fog / cognitive impairment

documentation as a clinical entity tag of type [neurological symptom

cluster / cognitive deficit] in the later neurological evaluation

records; (3) identification of this entity pair as a causal

candidate pair by the controlled vocabulary forced-tag system; (4)

retrieval of abstracts from the external database 106 supporting the

causal relationship between post-COVID-19 infection and neurological

cognitive sequelae; and (5) generation of a causal claim linking the

two findings, enriched with citations and attributed to the specific

source pages where each finding was documented.

The

cache-augmented multi-document causal extraction system 100

produced a structured report 108 containing: (1) a causal narrative

linking the COVID-19 infection history to the subsequent cognitive

symptoms, with the causal mechanism explained in terms consistent

with the supporting literature from the external database 106; (2)

page-level citations for the COVID-19 history documentation (source

document, page number); (3) page-level citations for the brain fog

symptom documentation (source document, page number); (4) abstract

citations supporting the biological plausibility of the causal

relationship; and (5) structured answers to a pre-defined set of

report questions covering longitudinal history, causal

relationships, objective metrics, and daily life impact.

A standard

RAG system presented with the same documents would fail to identify

this causal relationship because the COVID-19 history appears in

billing records and SOAP notes from Year 1, while the brain fog

symptom cluster appears in neurological evaluation notes from Year

4. A RAG system queried about neurological symptoms would retrieve

the Year 4 neurological evaluation notes as the most semantically

similar documents. It would not retrieve the Year 1 billing records,

because a billing record entry for a COVID-19 infection is not

semantically similar to a query about neurological symptoms. A

human investigator would almost certainly fail to identify this

causal relationship for similar reasons.

Notably, the

cache-augmented multi-document causal extraction system 100

identifies this relationship because: (a) the COVID-19 infection is

encoded as a causal-candidate precursor entity tag in the relational

cache at ingestion time; (b) the controlled vocabulary forced-tag

system identifies the COVID-19 → cognitive impairment pair as a

valid causal candidate; and (c) the large language model 104,

operating on the full relational cache, can traverse the causal

candidate relationship across the temporal and documentary gap

between the two findings.

This output

cannot be produced by prompting a generic LLM with the raw documents

102, because the causal candidate pair identification depends on the

relational structure of the relational cache, not on the

semantic content of any individual document. As mentioned above, a

human investigator would almost certainly fail to identify this

causal relationship for similar reasons.

This demonstrated example is merely

one way of using the cache-augmented multi-document causal

extraction system 100 in the medical

domain. This medical document intelligence system 100 ingests

various medical evidence or heterogeneous clinical records as a

batch of documents 102 including SOAP notes, EEG reports, MRI notes,

billing records, examination notes, including symptoms from patient

reports, diagnostic tests and imaging, observations from physician

notes, treatments and interventions, and outcomes measured by

results and progression.

The cache-augmented multi-document

causal extraction system 100 is configured to carry out

cross-temporal analysis by evaluating how much medical information

connects across time and across providers, mapping the evolution of

clinical findings through the full record to surface patterns that

span years, specialties, and documentation styles. The system 100 is

also configured to (1) carry out injury progression mapping that

reveals how injuries develop and are documented over time, (2)

identify critical inflection points in the clinical narrative, (3)

identify gaps in the clinical narrative, and (4) identify moments

where the record diverges from a claimed timeline. Using this

relational information, the system 100 is configured to detect

contradictions and inconsistencies within the medical evidence and

records. For example, conflicting diagnoses, divergent treatment

paths, and documentation gaps that may affect a case are all

examples of anomalies. Detection of these contradictions earlier

rather than later is key to correcting incorrect or incomplete

diagnoses and/or enabling enhanced preparation for cross-examination

in a legal proceeding.

Example Cross-Disciplinary Use

in Medical-Legal Matters

In some embodiments, an additional

layer is added to the prior example. In such an embodiment, the

cache-augmented multi-document causal extraction system 100 operates

at the intersection of legal and clinical evidence by transforming

medical records into structured evidence intelligence, rather than

chronological summaries to produce legally defensible causation

narratives with page-level cited provenance.

The system 100 employs a

combination of: (1) a custom extraction and vectorization pipeline

that encodes both clinical and legal entities with structural

document metadata; (2) a Cache-Augmented Generation (CAG)

architecture that operates on a full relational vector

representation of all documents simultaneously rather than

retrieving fragments; (3) MedGemma as the generative language model,

directed by engineered prompts to perform causal reasoning across

the cached relational structure; (4) a controlled vocabulary

forced-tag enrichment layer cross-referenced against PubMed

abstracts; and (5) a vector-space distance mapping system that

assigns page-level citations to every generated claim. The result is

a report that is structurally distinct from both human-authored

summaries and generic LLM outputs because it is cited, causal,

cross-referenced against published medical literature, and

defensible in a legal proceeding.

The cache-augmented multi-document

causal extraction system 100 is capable of identifying causal

relationships across disciplines because, in addition to the

embeddings encoding respective metadata and semantics, the

embeddings also encode the structural elements having assigned legal

entity tags. Additionally, the structural elements having assigned

medical entity tags. Accessible via the CAG architecture, the system

100 is able to identify links, connections, pairs, and causal

relationships amongst the entity tags and structural elements. These

embeddings, in combination with the engineered prompts and

MedGemma’s medical domain knowledge, render the system 100 capable

of identifying valid, medically plausible, and supported causal

candidate pairs among the documents.

The controlled vocabulary taxonomy

specific to this cross-disciplinary domain of legal and medical is

necessary in order to draw useful, significant, and valid

connections between the different domains. The taxonomy enables the

system 100 to link entity combinations as valid causal-candidate

pairs for forced-tag application. This legal-medical taxonomy

specific to TBI and PI cases was developed by a

board-certified functional neurologist (FACFN, FABBIR) and

encodes the domain expertise of a practicing specialist in

functional neurology and brain injury rehabilitation. The taxonomy

defines which clinical entity combinations are medically valid

causal precursor/sequela pairs for legal documentation purposes.

In other embodiments, the taxonomy

covers additional medical specialties beyond functional neurology

and TBI. The taxonomy is the bridge

between the relational structure of the vector cache and sources

outside of the original document upload. In this embodiment, the

source is PubMed which the system 100 uses to create a PubMed

enrichment layer. The taxonomy is not a publicly available or

derived from ontology and is provided by the cache-augmented

multi-document causal extraction system 100.

The system 100 applies the controlled vocabulary enrichment

layer to the initial output over the relational cache. This

vocabulary enrichment layer includes a predefined

taxonomy of medical-legal concepts, or “forced tags,” that are

systematically applied to the generated output to identify gaps and

opportunities for enrichment.

For each

causal claim in the generated output, the system 100: (1) extracts

the clinical entities involved in the causal relationship (e.g.

“COVID-19 infection” and “brain fog / cognitive impairment”);

(2) queries the PubMed database by retrieving abstracts matching the

identified clinical entity pair and causal relationship type; (3)

cross-references the retrieved PubMed abstracts against the causal

claim in the generated output; (4) enriches the output with

supporting literature citations where PubMed abstracts validate the

causal claim; and (5) flags causal claims for which no supporting

PubMed literature exists, reducing the risk of unsupported

assertions in the final report.

The

combination of semantic vector embeddings with simultaneously

encoded clinical entity tags and legal entity tags on structural

document elements (stored in a Qdrant relational vector database)

enables the system 100 to draw useful connections (relationships) to

produce a citation-supported report. This enables the system 100 to

encode both clinical and legal entity classes as first-class

relational properties of a vector cache for multi-document

medical-legal analysis. This dual encoding is what enables the

system 100 to answer queries that cross domains including clinical

and legal conceptual boundaries, e.g., identifying which clinical

findings (clinical entity) support which elements of a negligence

claim (legal entity).

Some

embodiments of the cache-augmented multi-document causal

extraction system 100 include fine-tuning

MedGemma on a specific dataset of medical-legal document pairs with

verified causal annotations, to improve causal identification

accuracy beyond the current base-model-plus-prompting approach.

The added

layer of computing vector-space distance between generated claims

and page-level cache entries to assign legally defensible page

citations enables the system 100 to generate sophisticated and

accurate medical-legal reports. The citation mechanism of the system

100 assigns citations based on semantic similarity in vector space,

producing citations that are grounded in meaning, rather than merely

surface form. This mechanism enables the cache-augmented

multi-document causal extraction system 100 to

generate reports where every claim is traceable to a specific page

in a specific source document, meeting the evidentiary standard

required for legal proceedings.

In some

embodiments, the cache-augmented multi-document causal

extraction system 100 applies CAG architecture

specifically to multi-document medical-legal analysis, where the

cache contains the full relational vector representation of

700–1,200 pages of heterogeneous medical records. The application

of CAG to these domains, with the specific relational structure

described, produces qualitatively different outputs from RAG-based

approaches, i.e., causal chains that span multiple documents and

years of patient history, rather than answers grounded in the most

similar retrieved fragment.

Before

we start, one important note: I'm not a patent attorney, so this is

technical and structural analysis of the claim language, not legal

That said, let's go claim by claim. I'll flag the issue, explain the risk, and suggest a fix for each.

Claim 1 — The Independent System Claim (The Most Important One)

Current language (key limitation):




"generate, for each structured text unit, a page-level vector representation encoding a semantic embedding of text content and one or more entity tags, the entity tags including causal-candidate precursors"

Issue 1 — "causal-candidate precursors" is undefined in the claim itself.

The spec defines it at paragraphs [0035] and [0062], but the claim

introduces the term without a self-contained definition. A competitor

could argue the claim is indefinite under 35 U.S.C. §112.

Suggested fix:




"...one

or more entity tags, each entity tag classifying the respective text

unit as belonging to a domain-specific entity class, at least one entity

tag being designated as a causal-candidate precursor indicating

membership in an entity class recognized by a controlled vocabulary as a

potential participant in a causal relationship"

Issue

2 — "controlled vocabulary" appears in the dependent claims but its

relationship to the entity tags in claim 1 is implicit, not explicit.

Claim 1 creates the relational cache and then validates against a

controlled vocabulary in the "generate a revised output" step — but the

claim doesn't establish that the controlled vocabulary was used to define

the entity tags at vectorization time. This gap means someone could

argue the claim reads on a system that applies any post-hoc filter, not

necessarily a pre-defined causal taxonomy.

Suggested fix: Add to the "generate vector representations" step:




"...wherein

the one or more entity tags are selected from a controlled vocabulary

of domain-specific entity classes defining valid causal precursor and

sequela class combinations"

Issue 3 — "perform inference and generate, by prompting a large language model operating on the relational cache"

The phrase "operating on the relational cache" is vague. Does the LLM

receive the full relational cache in context (CAG), or does it receive

outputs derived from querying the cache (which would read on standard

RAG)? This is the core technical distinction of your invention and it

needs to be explicit in claim 1.

Suggested fix:




"perform

inference by prompting a large language model with the full relational

cache loaded into the language model's context window simultaneously,

rather than with retrieved fragments, to generate a first output..."

Issue

4 — "determine, for each claim in the revised output, the respective

page-level vector representation having a highest semantic similarity to

a claim vector representation by mapping vector-space distances"

The word "claim" is doing double duty here — it means both a legal claim

(the patent document) and a factual assertion in the report. This is a

serious drafting ambiguity. In patent prosecution, examiners read claims

very literally.

Suggested fix: Replace "claim" with "assertion" throughout this limitation:




"determine,

for each assertion in the revised output, the respective page-level

vector representation having the highest semantic similarity to an

assertion embedding..."

Claim 2 — Document Type Enumeration

Current language:




"SOAP notes, EEG reports, MRI notes, billing records, examination notes, unstructured PDF records"

Issue — This list is illustrative but the "one or more of" language makes it almost infinitely broad. Any system processing any document type would satisfy this. It adds no meaningful limitation.

Suggested fix: Either delete claim 2 entirely (it adds nothing defensible) or convert it to reinforce the metadata-preservation aspect:




"wherein

the extraction pipeline preserves document-type classification metadata

for each structured text unit, distinguishing between at least clinical

document types and billing document types"

That language is narrower but actually more defensible because it captures the technical feature that matters — dual-domain metadata classification.

Claim 3 — Dual Entity Tag Encoding

This is one of your strongest claims and should arguably be pulled up into claim 1 or made an independent claim.

Current language:




"the

clinical entity tag encodes at least one of a symptom, a diagnosis, a

medication, a diagnostic finding, an anatomical reference, and a

temporal marker, and wherein the legal entity tag encodes at least one

of an injury event, a causation claim, a functional limitation, and a

daily life impact descriptor"

Issue — The "at least one of" construction is a disjunctive list. Under current USPTO guidance, "at least one of A, B, and C" means "at least one of {A, B, C}" — so a vector encoding only a temporal marker satisfies the clinical entity tag limitation. That's not your invention.

Suggested fix:




"wherein

the clinical entity tag encodes a clinical entity class selected from

the group consisting of: symptoms, diagnoses, medications, diagnostic

findings, anatomical references, and temporal markers; and wherein the

legal entity tag encodes a legal entity class selected from the group

consisting of: injury events, causation claims, functional limitations,

and daily life impact descriptors; and wherein each vector

representation simultaneously encodes at least one clinical entity tag

and at least one legal entity tag"

The

system from anything in prior art. It must be in the claim.

Issue — Claiming a specific commercial product (Qdrant) is a significant vulnerability.

If Qdrant changes its architecture, gets acquired, or if a competitor

characteristics, they avoid the claim entirely. This is a classic

"wherein

representation as a structured record comprising a vector field, a

metadata payload field, and one or more relational property fields

encoding relationships to other vector representations in the database"

That reads on Qdrant, Milvus, Weaviate, pgvector, and any future equivalent — which is what you want.

"...cross-modal

"wherein
